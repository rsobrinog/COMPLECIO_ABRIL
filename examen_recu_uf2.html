Configuració prèvia — Docker

Passos inicials

  • Crea el repositori a GitHub (privat) amb README.md
  • Clona'l i crea el fitxer docker-compose.yml en una carpeta
  • Engega els serveis amb docker compose up -d
  • Comprova que la BD és accessible via pgAdmin
  • Crea la taula clients a la BD comp_DB

Comandes Docker útils

  • sudo docker compose up -d — engegar serveis
  • sudo docker ps — veure contenidors actius
  • sudo docker stop <nom> — parar contenidor
  • sudo docker rm <nom> — eliminar contenidor
  • sudo lsof -i:5432 — alliberar port si cal

docker-compose.yml

version: '3.1'
services:
  db:
    image: postgres:13
    container_name: db_complecio
    environment:
      - POSTGRES_DB=comp_DB
      - POSTGRES_PASSWORD=admin
      - POSTGRES_USER=admin
    ports:
      - "5432:5432"
    volumes:
      - local_pgdata:/var/lib/postgresql/data
  pgadmin:
    image: dpage/pgadmin4
    container_name: pg_complecio

      - "80:80"
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@comp.com

Connexió a la base de dades
Crear el fitxer de connexió a PostgreSQL
10%

Crea la carpeta i el fitxer de connexió a la base de dades PostgreSQL seguint la configuració del docker-compose proporcionat.

El fitxer ha de contenir una funció que estableixi la connexió i que retorni únicament l'objecte de connexió. Aquesta funció s'utilitzarà en tots els endpoints que necessitin accés a la BD.

💡
Paràmetres de connexió: host = localhost, database = comp_DB, user = admin, password = admin, port = 5432
Criteris d'avaluació
S'ha creat una carpeta específica per a la connexió (p.ex. db_connect/)
Dintre de la carpeta hi ha el fitxer de connexió
La connexió està encapsulada dins d'una funció
La funció retorna únicament l'objecte de connexió
Les credencials coincideixen amb el servei Docker del projecte

3
Consulta del perfil de client (GET)
Retornar dades del client filtrant camps sensibles
20%

El frontend necessita mostrar la informació del perfil d'un client concret. Crea un endpoint GET que, donat l'identificador del client, retorni les seves dades.

Important: la resposta no ha d'incloure els camps: password, descripcio ni cp (codi postal). Aquests camps mai han d'arribar al frontend.

⚠️
El response_model d'un endpoint GET que retorna una llista ha de ser List[dict] o el tipus corresponent. No si retorna un únic element.
Criteris d'avaluació
Endpoint

S'especifica el response_model correcte
Funció asíncrona
Es passa la dada necessària per identificar el client
Es crida al servei adequat
Es retorna la resposta serialitzada (via schema, que es crea a l'Ex.4)
Servei
La query SELECT obté les dades del client correctament
El return passa les dades en brut (raw) al endpoint, que serà l'schema qui les filtra

4
Schema de serialització
Transformar dades de la BD a format JSON per al frontend
15%

El projecte necessita una capa que transformi les files de qualsevol consulta de lectura (retornades per la BD com a tuples) en un format JSON que el frontend pugui interpretar.

Crea un schema amb dues funcions:

  • Una funció que mapeja una sola fila de la BD a un diccionari Python amb els camps adequats (excloent els camps sensibles: password, descripcio, cp)
  • Una funció que itera sobre la llista de files i crida la funció anterior per a cada element

🚫
Atenció crítica: Si el schema existeix però no s'utilitza als endpoints de lectura, no puntua (0 punts per l'exercici 4 sencer).
# Exemple d'estructura esperada (schemas/client_schema.py)

def client_schema(client) -> dict:
    return {
        "id": client[0],
        "nom": client[1],
        "cognom": client[2],
        "adreca": client[3],
        # cp, descripcio i password NO s'inclouen
    }

def clients_schema(clients) -> list:
    return [client_schema(client) for client in clients]
Criteris d'avaluació
Es crea la carpeta schemas/
Es crea el fitxer de l'schema dins de la carpeta
S'implementa la funció per mapejar una sola fila a diccionari (sense camps sensibles)
S'implementa la funció per iterar una llista de files i aplicar la funció anterior
L'schema s'utilitza correctament als endpoints que fan lectures (Ex.3, Ex.5...)

Modificació de dades del perfil (PUT)
Actualitzar cognom i adreça d'un client
20%

El frontend permet als clients modificar parcialment el seu perfil. Crea un endpoint PUT que actualitzi únicament els camps cognom i adreca del client que fa la petició.

El client s'identifica a través de les dades que arriben a la petició (no cal passar l'id pel path de la URL).

6
Eliminació del perfil (DELETE)
Esborrar el registre del client de la BD
10%

El frontend permet eliminar el compte del client. Crea un endpoint DELETE que esborri el registre del client actual de la taula clients.

El return de l'endpoint ha de ser un diccionari (mai un booleà directament). La resposta JSON ja ve serialitzada des del servei: no cal transformar-la al return de l'endpoint.

🚫
El servei no ha de retornar un booleà. L'endpoint tampoc. Retorna sempre un diccionari com ara {"deleted": True} o les dades del registre eliminat.
Criteris d'avaluació
Endpoint

El path és adequat per a una eliminació
S'especifica el response_model correcte
Funció asíncrona

Es crida al servei adequat
El return de l'endpoint és la resposta del servei (sense transformar-la de nou)
Servei
La query DELETE elimina el registre correctament
El servei retorna un diccionari (p.ex. {"deleted": True}), mai un booleà sol
